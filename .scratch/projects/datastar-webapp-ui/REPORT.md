## 1. System Snapshot — What Exists Today

### 1.1 Companion Sidebar: Current Neovim Flow

The current companion sidebar lives entirely inside Neovim. The flow is:

```
Cursor moves onto a Python/Markdown/TOML symbol
  → Neovim sends CursorHold → $/remora/cursorMoved notification to LSP server
  → LSP server emits CursorFocusEvent on EventBus
  → NodeAgentRouter._on_cursor_focus() picks up the event
  → NodeAgentRegistry.get_or_create(node) lazy-loads a NodeAgent
  → NodeAgent.on_cursor_focus() calls compose_sidebar()
  → NodeAgentSidebarReady event emitted on EventBus
  → LSP server catches it, sends $/remora/companionSidebarUpdated to Neovim
  → Lua panel.on_event() re-renders the sidebar buffer
```

The sidebar content is generated by `compose_sidebar()` (`companion/sidebar/composer.py`), which assembles markdown from:
- Node identity (name, type, file:line)
- User notes (`notes/user_notes.md` in workspace)
- Agent observations (`notes/agent_notes.md`)
- Workspace panels: role.md, schema.yaml, notes.md, summary.md, todo.md, log.jsonl, tools/ directory
- Recent conversation index (last 5 sessions, summarized)
- Connections/links (via LinksResolver from `links/links.json`)

Chat is handled via LSP commands: `companion.sendMessage` → `NodeAgent.send()` → LLM kernel → response. There are also commands for notes, links, and history.

**The Neovim dependency:** The entire interaction model relies on the Neovim LSP protocol (JSON-RPC over stdio) and the Lua panel implementation. Every UI concern (rendering, input, layout) is in Lua.

### 1.2 Existing Starlette + Datastar Infrastructure

There is already a working Starlette HTTP server (`adapters/starlette.py`) with these routes:

| Route | Purpose |
|-------|---------|
| `GET /` | HTML dashboard (renders via `render_index_html`) |
| `GET /subscribe` | Datastar SSE stream — patches the entire UI on each event |
| `GET /events` | Raw SSE stream of all EventBus events |
| `GET /replay` | Historical event replay with optional follow mode |
| `POST /input` | Submit human input responses |
| `GET /config` | Config snapshot |
| `GET /snapshot` | UI state snapshot |
| `GET /swarm/agents` | List all agents |
| `GET /swarm/agents/{id}` | Get one agent |
| `POST /swarm/events` | Emit event to swarm |
| `GET /swarm/subscriptions/{id}` | Agent subscriptions |

The `service/datastar.py` module wraps the `datastar_py` library (using `ServerSentEventGenerator`) for `patch_elements` (HTML patches) and `patch_signals` (JS signal patches). The HTML shell already loads Datastar v1.0.0-RC.7 from CDN.

The existing dashboard currently shows: event stream, blocked agents, agent status, progress bar, and a graph launcher.

### 1.3 Graph Data Model

Nodes are `AgentNode` Pydantic models stored in the `nodes` table (in EventStore's SQLite DB). Key fields for visualization:
- `node_id` (deterministic hash, e.g. `mymodule.MyClass.my_method`)
- `node_type`: `function`, `class`, `method`, `file`, `section`, `table`
- `name`, `full_name`, `file_path`, `start_line`, `end_line`
- `status`: `idle`, `running`, `error`, `pending_approval`
- `caller_ids`, `callee_ids` (lists of node_ids)
- `parent_id` (parent node in the code hierarchy)

Edges live in the RemoraDB `edges` table with types: `parent_of`, `calls`.

`NodeStore.list_nodes()` returns all nodes; `read_graph()` supports node lookup and neighbor traversal. `LazyGraph` wraps this with a `rustworkx.PyDiGraph` for in-memory graph algorithms.

A medium Python project yields ~100–300 nodes and ~300–1000 edges (this project: 106 nodes, 305 edges).

### 1.4 EventBus Streaming Capabilities

`EventBus.stream()` is an async context manager that yields events from an in-memory asyncio queue. It is already used in `RemoraService.subscribe_stream()` and `events_stream()`. Every emitted event (`NodeDiscoveredEvent`, `AgentStartEvent`, `AgentCompleteEvent`, `HumanChatEvent`, `NodeAgentSidebarReady`, etc.) is available on this stream.

`render_event_sse()` in `service/handlers.py` normalizes any event to a JSON SSE frame with `event_type`, `agent_id`, `summary`, `payload`, and `timestamp`. This is the exact data needed for the live event log and graph pings.

### 1.5 NodeAgent / NodeAgentRegistry

`NodeAgentRegistry` maintains a live pool of `NodeAgent` instances (LRU, max configurable). Each `NodeAgent`:
- Has a persistent Cairn workspace (files: role.md, schema.yaml, notes.md, summary.md, todo.md, log.jsonl, tools/, chat/, inbox/, guides/)
- Exposes `send(content)` for chat (calls the LLM kernel)
- Exposes `on_cursor_focus()`, `on_content_changed()`, `on_file_saved()`, `on_inter_agent_message()`
- The registry resolves agents lazily: `get_or_create(node)` creates one on first access

The companion handler commands (`companion.getSidebar`, `companion.sendMessage`, etc.) are currently LSP command handlers. For the web app, these need HTTP equivalents.

## 2. Key Integration Points

### 2.1 REST + SSE Endpoints Already Present

The Starlette adapter gives us:
- **`GET /events`** — live SSE of all EventBus events, normalized. Backbone for the event log sidebar and graph ping system.
- **`GET /swarm/agents`** — full list of AgentNodes including `status`, `node_type`, `caller_ids`, `callee_ids`.
- **`GET /swarm/agents/{id}`** — single node data.
- **`POST /swarm/events`** — emit `AgentMessageEvent` or `ContentChangedEvent` to trigger agents.

What is **missing** for the web UI:
- Full graph topology (nodes + edges) as a single JSON payload
- Node chat endpoint (`POST /companion/chat`)
- Node workspace data (`GET /companion/workspace/{node_id}`)
- Node-filtered SSE stream

### 2.2 NodeStore: list_nodes, get_node, read_graph

`event_store.nodes.list_nodes()` can fetch all nodes without source code (using `columns=` filter to avoid large blobs). This is the graph bootstrap call — invoked once on page load, then kept live via SSE events.

### 2.3 EventBus.stream() as the Live Feed

The live event feed drives two concerns simultaneously:
1. **Event log sidebar** — append every event with timestamp, type, summary
2. **Graph node pings** — when `agent_id` is present, highlight that node

Every agent-lifecycle event carries `agent_id`. Every interaction event carries `from_agent`/`to_agent`. We know exactly which node(s) to ping for every event.

### 2.4 compose_sidebar() as the Data Source for Mode 1

`compose_sidebar(node, workspace)` already produces all content for the node panel. In the web context, instead of rendering as a Neovim markdown buffer, we return it as markdown for browser-side rendering (e.g. via marked.js) or decompose it into structured JSON sections.

### 2.5 LazyGraph + Edge Types

For visualization, we don't need the rustworkx wrapper — D3/Cytoscape/Sigma consume flat node/edge arrays. We obtain these via direct DB queries: `NodeStore.list_nodes()` for nodes, and an edges query to RemoraDB.

Edge types to expose:
- `parent_of` — hierarchical containment (file → class → method)
- `calls` — function call relationship

These two edge types get distinct visual styles (dashed vs solid, different colors).

## 3. Proposed UI Architecture — Two Modes

### 3.1 Mode 1: Local Agent Node UI (Narrow Sidebar)

A tall, narrow panel (~280–360px wide). Shows the currently-focused agent node in detail:

**Top — Node identity:**
- Node name (large heading)
- Node type badge (function/class/method/file)
- File path and line range
- Status indicator (idle/running/error/pending_approval) with color

**Middle — Workspace panels (tabbed or scrolled):**
- Notes (user_notes.md / agent_notes.md)
- Role & Schema
- Todo
- Recent activity log (last 20 lines of log.jsonl)
- Available tools (.pym files in tools/)
- Connections (clickable links → navigate to that node's panel)

**Bottom — Chat interface:**
- Scrollable conversation history
- Text input with submit button
- Live streaming responses

**Data needs:**
- `GET /companion/sidebar/{node_id}` → markdown + workspace data
- `POST /companion/chat` with `{node_id, message}` → streaming response
- `GET /companion/workspace/{node_id}` → file tree
- `POST /companion/workspace/{node_id}/note` → append note
- SSE subscription filtered to `agent_id == node_id`

### 3.2 Mode 2: Live Graph View (Fullscreen)

Fills the browser window with two sub-panels:

**Main area (75–80% width): Graph canvas**
- Force-directed or hierarchical graph of all AgentNodes
- Nodes colored by `node_type` (class=blue, function=green, method=teal, file=orange)
- Nodes sized by connectivity (degree) or recent event count
- Status ring/border: green=idle, blue=running, yellow=pending_approval, red=error
- Edge styles: solid=calls, dashed/thin=parent_of
- **Live ping animation:** when an event fires for a node, that node pulses (size burst + color flash, 500–1000ms, then reverts)
- Click any node → opens Mode 1 panel as a slide-in drawer
- Hover → tooltip with node_id, type, last event summary

**Right sidebar (20–25% width): Live event log**
- Scrolling list of recent events (newest first, auto-scrolling)
- Each entry: timestamp, event type badge (colored), agent name, summary
- Clicking an entry → highlights the corresponding node in the graph
- Filter controls: by event type, by agent

**Controls strip:**
- Search/filter nodes by name/type/file
- Layout toggle (force-directed vs hierarchical)
- Pause/resume live updates
- Zoom to fit

### 3.3 Mode Transition & Routing Strategy

Three options:

**Option 1 — URL hash routing:** `/#/graph` (Mode 2), `/#/node/{id}` (Mode 1 panel active). Clean, bookmarkable.

**Option 2 — Panel overlay (recommended):** Mode 2 is always the base view. Mode 1 opens as a slide-in drawer from the right, the graph shrinks to ~65% width. Clicking the graph background collapses the panel. Optional URL history.pushState updates.

**Option 3 — Separate pages:** `/` → Mode 2, `/node/{id}` → Mode 1 fullscreen. Cleanest separation, easiest to deep-link.

## 4. Option A — Datastar Signals + D3.js Force Graph

### Overview

Use Datastar's signal system as the state layer and D3.js force simulation for graph rendering.

### Architecture

D3 renders an SVG (or canvas) force graph. Node and link data are updated via `simulation.nodes()` and `simulation.force("link").links()`. Datastar signals hold `nodeStates[id].status` and `nodeStates[id].pinging`. D3 reads these each tick.

Ping animation via D3 transitions:
```js
function triggerNodePing(nodeId) {
  const node = nodeCircles.filter(d => d.id === nodeId);
  node.transition().duration(200).attr("r", r => r * 2)
      .transition().duration(300).attr("r", r => r);
  node.attr("class", "pinging").transition().delay(500).attr("class", "");
}
```

### Pros
- Maximum customization of every visual element
- D3 is the de-facto standard; extensive documentation and examples
- SVG is crisp at all zoom levels
- Force simulation produces natural organic layouts
- Excellent transition/animation API
- Semantic edge styling (dashed parent_of vs directional calls arrows) trivial

### Cons
- Steep learning curve — D3 requires understanding data-joins, enter/update/exit
- Integrating D3 state with Datastar signals requires custom bridging code
- SVG performance degrades above ~500–1000 nodes
- Large bundle (~80KB D3 core, more with plugins)
- Force simulation for large graphs needs careful tuning

### Implication
D3 gives the most visual control but carries the most implementation cost. Best if bespoke aesthetics are a priority.

## 5. Option B — Datastar Signals + Cytoscape.js

### Overview

Cytoscape.js is purpose-built for graph/network visualization. It handles layout, interaction, styling, and animation out of the box.

### Architecture

```js
const cy = cytoscape({
  container: document.getElementById("graph"),
  elements: { nodes: [...], edges: [...] },
  style: [
    { selector: "node[type='function']", style: { "background-color": "#a6e3a1" }},
    { selector: "node[status='running']", style: { "border-width": 3, "border-color": "#89b4fa" }},
    { selector: ".pinging", style: { "border-width": 6, "border-color": "#f9e2af" }},
    { selector: "edge[type='calls']", style: { "line-color": "#89b4fa", "target-arrow-shape": "triangle" }},
    { selector: "edge[type='parent_of']", style: { "line-style": "dashed", "line-color": "#585b70" }},
  ],
  layout: { name: "cose" }
});

// SSE ping integration
eventSource.onmessage = (e) => {
  const ev = JSON.parse(e.data);
  if (ev.agent_id) {
    cy.getElementById(ev.agent_id).flashClass("pinging", 800);
    cy.getElementById(ev.agent_id).data("status", ev.new_status || "running");
  }
};

// Mode 1 trigger
cy.on("tap", "node", (e) => {
  const nodeId = e.target.id();
  openNodePanel(nodeId);
});
```

### Pros
- Purpose-built for graph use cases — all interaction primitives built in
- `flashClass()` is exactly what's needed for event pings
- Simpler API than D3 for graph-specific operations
- ~170KB gzipped (much lighter than vis.js)
- CSS-like stylesheet system is intuitive
- Plugin ecosystem: `cytoscape-dagre` (hierarchical), `cytoscape-cose-bilkent` (fast force), `cytoscape-node-html-label` (rich labels)
- Canvas rendering (better performance than SVG for medium graphs)
- Easy dynamic add/remove/update of nodes

### Cons
- Less flexible than D3 for completely custom rendering
- Stylesheet DSL slightly different from standard CSS
- Needs manual Datastar signal bridging for the sidebar (graph canvas remains Cytoscape's domain)
- `cose` layout slow for >300 nodes without the CoSE-Bilkent plugin

### Opportunity
Cytoscape supports **compound nodes** — a file node can contain its class and function children as collapsible sub-graphs. This gives the graph a natural hierarchy matching the code structure: files collapse/expand to show their contents. This is a significant UX feature not trivially available in other options.

## 6. Option C — Datastar Signals + Sigma.js + Graphology

### Overview

Sigma.js uses WebGL for rendering, making it the fastest option for large graphs (1,000+ nodes). Graphology is its companion data model.

### Architecture

```js
const graph = new graphology.Graph({ type: "directed" });
// populate from API

const renderer = new Sigma(graph, container, {
  nodeReducer: (node, data) => ({
    ...data,
    color: statusColor(data.status),
    size: data.degree * 2 + 5,
  }),
});

// Ping via attribute mutation + refresh
eventSource.onmessage = (e) => {
  const { agent_id } = JSON.parse(e.data);
  const origSize = graph.getNodeAttribute(agent_id, "size");
  graph.setNodeAttribute(agent_id, "size", origSize * 2.5);
  graph.setNodeAttribute(agent_id, "highlighted", true);
  renderer.refresh();
  setTimeout(() => {
    graph.setNodeAttribute(agent_id, "size", origSize);
    graph.setNodeAttribute(agent_id, "highlighted", false);
    renderer.refresh();
  }, 800);
};
```

### Pros
- WebGL rendering: handles 10,000+ nodes smoothly
- Graphology is a first-class JS graph library with algorithms (centrality, shortest path, clustering)
- Future-proof for large monorepos
- Active 2024–2025 maintenance

### Cons
- Overkill for current scale (100–300 nodes)
- WebGL debugging is harder than SVG/canvas
- Less visual customizability than D3
- No built-in hierarchical layout (requires plugin or custom force)
- Sigma's node click API is slightly more verbose than Cytoscape's

### Opportunity
Graphology algorithms could drive interesting views: "top 10 most central functions", "nodes with highest betweenness centrality" (hotspots in the call graph). These analysis features are natural extensions once the graph is rendered.

## 7. Option D — Datastar Signals + vis.js Network

### Overview

vis.js Network is a mature, batteries-included graph library with physics simulation and DataSet-based dynamic updates.

### Architecture

```js
const nodesDataSet = new vis.DataSet(nodes);
const edgesDataSet = new vis.DataSet(edges);
const network = new vis.Network(container, { nodes: nodesDataSet, edges: edgesDataSet }, {
  physics: { enabled: true, stabilization: { iterations: 100 } },
});

network.on("click", (params) => {
  if (params.nodes.length > 0) openNodePanel(params.nodes[0]);
});

function ping(nodeId, eventType) {
  const orig = nodesDataSet.get(nodeId);
  const pingColor = eventColorMap[eventType] || "#f9e2af";
  nodesDataSet.update({ id: nodeId, borderWidth: 5, color: { border: pingColor } });
  setTimeout(() => nodesDataSet.update({ id: nodeId, ...originalStyle(orig) }), 800);
}
```

### Pros
- Very mature, large community, many examples
- DataSet API makes dynamic node/edge updates trivial
- Built-in physics for organic layout
- Click, hover, drag all built in
- Built-in hierarchical layout (`layout: { hierarchical: { direction: "UD" } }`)

### Cons
- Large bundle (~1MB uncompressed) — largest of all options
- Canvas rendering: custom visual elements harder than SVG
- Less actively maintained in recent years vs Cytoscape/Sigma
- Physics stabilization can be slow/jittery for larger graphs
- Less visually polished default aesthetics

### Opportunity
vis.js hierarchical layout trivially maps `parent_of` edges to a top-down tree (file → class → method). This requires zero layout configuration — just enable hierarchical mode.

## 8. Option E — Datastar-Only (Inline SVG, Server-Computed Layout)

### Overview

No JavaScript graph library. The server computes a layout once via Python's NetworkX and returns SVG-ready `x,y` coordinates. The browser renders a static SVG with Datastar-driven updates.

### Architecture

```python
# Server: compute layout, cache until topology changes
import networkx as nx

def compute_layout(nodes, edges):
    G = nx.DiGraph()
    for n in nodes: G.add_node(n["node_id"])
    for e in edges: G.add_edge(e["from"], e["to"])
    pos = nx.spring_layout(G, seed=42, k=2.5)
    return {nid: {"x": x * 500 + 500, "y": y * 400 + 400}
            for nid, (x, y) in pos.items()}
```

```html
<svg data-signals="{nodeStates: {}}">
  <circle
    id="node-my_func"
    cx="320" cy="180"
    data-class="{'pinging': $nodeStates['my_func']?.pinging}"
    data-on-click="$selectedNode = 'my_func'"
  />
</svg>
```

```css
@keyframes ping { 0% { r: 10; } 50% { r: 22; opacity: 0.5; } 100% { r: 10; } }
.pinging { animation: ping 0.8s ease-out; }
```

### Pros
- No additional JS libraries — pure Datastar + server-computed layout
- Fully server-side graph logic (fits Datastar's hypermedia philosophy)
- Simplest possible client code — fastest to build a working PoC (~1 day)
- Layout is stable and deterministic between page loads
- Zero JS bundle overhead

### Cons
- No interactive physics — users can't drag-to-reorganize
- NetworkX spring layout is slow for >200 nodes (recalculate on topology change is expensive)
- No smooth zoom/pan without additional CSS/JS transforms
- Less visually engaging than a live force simulation
- SVG performance still degrades above ~500 nodes
- Harder to add compound node grouping later

### Opportunity
This option is the fastest path to a proof-of-concept. The layout can be cached between runs and only invalidated when `NodeDiscoveredEvent` or `NodeRemovedEvent` fires. For an initial investigation/demo, a working visualization is achievable in one day.

## 9. Server-Side Architecture Options

### 9.1 Extend Existing Starlette Adapter In-Place

Add new routes directly to `adapters/starlette.py`. Minimal structural change. New routes needed:

| Route | Method | Purpose |
|-------|--------|---------|
| `/graph/data` | GET | All nodes + edges as JSON (no source blobs) |
| `/companion/sidebar/{node_id}` | GET | Sidebar markdown + structured workspace data |
| `/companion/chat` | POST | Send message to node agent, get response |
| `/companion/chat/{node_id}/stream` | GET | SSE stream for streaming response tokens |
| `/companion/workspace/{node_id}` | GET | Workspace file tree |
| `/companion/workspace/{node_id}/note` | POST | Append user note |

`RemoraService` gains a `companion_registry` attribute (same as the LSP server). The registry requires `CairnWorkspaceService` and `EventBus`, both already available on `RemoraService`.

**Pros:** Least structural change, one Starlette app to run and configure.
**Cons:** `starlette.py` grows large; companion and swarm concerns coexist.

### 9.2 New GraphWebService Alongside RemoraService

Create `service/graph_web_service.py` wrapping `RemoraService` with graph-specific capabilities:

```python
class GraphWebService:
    def __init__(self, base: RemoraService, registry: NodeAgentRegistry):
        self._base = base
        self._registry = registry

    async def graph_data(self) -> dict:
        nodes = await self._base._event_store.nodes.list_nodes(
            columns=["node_id", "node_type", "name", "full_name",
                     "file_path", "start_line", "end_line", "status",
                     "caller_ids", "callee_ids", "parent_id"]
        )
        edges = await self._fetch_edges()
        return {"nodes": [n.model_dump() for n in nodes], "edges": edges}
```

**Pros:** Clean separation; companion logic not mixed with core swarm API.
**Cons:** More abstraction layers; `GraphWebService` may just re-expose things from `RemoraService`.

### 9.3 Separate Companion HTTP Server Process

The companion system could run as an independent Starlette app on a different port. It receives events via the shared in-process EventBus, or mounts as a sub-application using Starlette's `Mount`.

**Pros:** Complete isolation; companion can be started/stopped independently.
**Cons:** Inter-process communication complexity if truly separate processes.

**Recommendation:** Option 9.1 for initial implementation. Refactor to 9.2 once the API surface stabilizes.

## 10. Live Ping / Animation Strategy

### 10.1 Datastar Signal Injection Per Node (Server-driven)

On each event with an `agent_id`, the server also pushes a `patch_signals` frame:

```python
async def graph_events_stream(event_bus):
    async with event_bus.stream() as events:
        async for event in events:
            yield render_event_sse(event)
            agent_id = getattr(event, "agent_id", None)
            if agent_id:
                safe_id = agent_id.replace(".", "_").replace("/", "_")
                yield SSE.patch_signals({
                    f"nodeState_{safe_id}": {
                        "pinging": True,
                        "event_type": type(event).__name__,
                        "ts": time.time(),
                    }
                })
```

The graph JS subscribes to signal changes and triggers animations. Pins are cleared by client-side timer.

**Pros:** Datastar-idiomatic; server controls all state.
**Cons:** Intermixes HTML patch events with graph control events in the same stream.

### 10.2 Client-Side SSE Handler → Graph Library API (Recommended)

The client subscribes to `/events` directly (not via Datastar) and calls the graph library's API:

```js
const eventColorMap = {
  "AgentStartEvent":    "#89b4fa",  // blue
  "AgentCompleteEvent": "#a6e3a1",  // green
  "AgentErrorEvent":    "#f38ba8",  // red
  "HumanChatEvent":     "#cba6f7",  // purple
  "RewriteProposalEvent": "#fab387", // orange
};

const sse = new EventSource("/events");
sse.onmessage = (e) => {
  const event = JSON.parse(e.data);
  appendToEventLog(event);
  if (event.agent_id) {
    pingNode(event.agent_id, event.event_type);
  }
};

// Cytoscape:
function pingNode(nodeId, eventType) {
  const color = eventColorMap[eventType] || "#f9e2af";
  const el = cy.getElementById(nodeId);
  el.style("border-color", color).style("border-width", 5);
  el.animate({ style: { "border-width": 5 } }, { duration: 200 })
    .animate({ style: { "border-width": 1 } }, { duration: 600 });
}
```

This is the cleanest approach: Datastar handles the sidebar/mode-1 concerns, the graph canvas remains the graph library's domain.

### 10.3 CSS Animation via data-class Binding (Option E only)

For inline SVG (Option E), Datastar can bind CSS classes directly to node elements, driven by signal values. Limited to the most recent event's node (concurrent pings require careful signal management).

## 11. Risks, Opportunities, and Open Questions

### Risks

**R1 — Graph scale.** At 100–300 nodes, all options work smoothly. At 1,000+ nodes (large monorepo), only Sigma.js (WebGL) maintains performance. The choice made now may need revisiting.

**R2 — Companion registry lifecycle.** `NodeAgentRegistry` is currently created inside the LSP server. Wiring it into the Starlette adapter requires ensuring `CairnWorkspaceService` and `EventBus` are correctly shared. Running both the LSP server and Starlette webapp in the same process (or sharing an EventBus across processes) needs explicit design.

**R3 — Chat streaming.** `NodeAgent.send()` awaits completion before returning. For the web UI, users expect streaming tokens as the LLM thinks. This requires verifying whether `structured_agents`' kernel supports token streaming and adding a streaming-aware HTTP endpoint.

**R4 — Concurrent workspace writes.** Multiple web clients opening the same node's panel share the same workspace. Chat uses session-scoped files (safe). Notes use `append_text` (safe). Schema/role edits would need locking.

**R5 — Graph layout stability.** Force-directed layouts produce different positions each run. Node positions should be persisted (localStorage or SQLite) so the graph doesn't reshuffle on every page load. This is important for user spatial memory.

**R6 — Cross-origin.** If the LSP server (stdio) and Starlette webapp run in separate OS processes, the Neovim cursor events won't reach the web graph automatically. The webapp and LSP server should share an in-process EventBus or use a lightweight pub/sub bridge.

### Opportunities

**O1 — Cursor-graph sync.** The LSP server already emits `CursorFocusEvent` on `CursorHold`. The web graph picks this up via SSE and highlights the cursor-focused node in real time. The developer sees their position in the code reflected on the graph as they edit.

**O2 — Bidirectional graph-to-editor.** When the user clicks a node in the web graph, post an event that triggers the Neovim cursor to jump to that file:line. Requires a reverse channel (HTTP endpoint on LSP server or shared EventBus event).

**O3 — Subgraph views.** Filter the graph: "show only nodes in this file", "show nodes by status=running", "show the full call chain from this node". Trivial with Cytoscape's built-in filter API.

**O4 — Event replay scrubber.** `EventStore.replay()` with `after_id`/`since` parameters enables a timeline scrubber in Mode 2: drag a slider to replay events from a past session and watch the graph animate through historical activity. Powerful for debugging agent cascades.

**O5 — Live diff viewer.** When a `RewriteProposalEvent` fires, Mode 1 can show the diff inline with syntax highlighting. Accept/reject buttons post to `/input`.

**O6 — Mobile node panel.** Mode 1 functions as a standalone mobile app — a focused agent chat interface reachable from any device, removing the Neovim requirement entirely.

**O7 — Codebase onboarding tool.** The live graph with agent status shows which parts are being actively analyzed/rewritten. This is a powerful mental model builder for users new to a codebase — nothing like it exists in current tooling.

### Open Questions

**Q1:** Should the webapp run inside the same Starlette app as the existing dashboard, or as a new mount point (e.g. `/ui/` prefix)?

**Q2:** Should `NodeAgentRegistry` be initialized in the Starlette adapter, or only in the LSP server? If the LSP server is not running, can the webapp still provide read-only graph views?

**Q3:** Does the `structured_agents` kernel support token-streaming? If not, is polling the workspace's `log.jsonl` a viable fallback for chat progress?

**Q4:** How should graph layout be persisted? `localStorage` per browser? SQLite per workspace? Per-user session?

**Q5:** What is the target event-to-visual-ping latency? Under 200ms requires WebSocket; under 500ms is achievable with SSE (current approach).

**Q6:** Should Mode 2 show all nodes by default, or only a viewport-based lazy-loaded neighborhood? Large monorepos may need paging.

## 12. Recommendation Summary

### Primary Recommendation: Option B — Datastar Signals + Cytoscape.js

**Rationale:** Cytoscape.js is purpose-built for the graph/network visualization use case. Its API surface exactly matches what's needed:
- `flashClass()` for event ping animations
- CSS-like stylesheets for type/status differentiation  
- `cose-bilkent` plugin for fast force-directed layout
- Compound nodes for file→class→method collapsible groups
- `tap` event for node click → Mode 1 panel
- Canvas rendering for smooth performance at current scale (100–300 nodes)
- ~170KB gzipped — much lighter than vis.js (1MB)

For Remora's current scale, Cytoscape performs smoothly and offers the best balance of capability, simplicity, and visual quality. If the codebase scales to 1000+ nodes, migrating the canvas renderer to Sigma.js is possible while keeping the same interaction logic.

### Server-Side: Option 9.1 (Extend Starlette Adapter)

Extend `adapters/starlette.py` and `service/api.py`:
1. `GET /graph/data` — nodes + edges, no source blobs
2. `GET /companion/sidebar/{node_id}` — sidebar data (markdown or JSON)
3. `POST /companion/chat` — send message, synchronous response
4. `GET /companion/chat/{node_id}/stream` — SSE for streaming response (Phase 3)
5. `GET /companion/workspace/{node_id}` — file tree
6. `POST /companion/workspace/{node_id}/note` — append note

`RemoraService` gains a `companion_registry` attribute wired at startup.

### Live Updates: Option 10.2 (Client SSE → Cytoscape API)

Browser opens a direct `EventSource("/events")`. Each event appends to the log sidebar. If `agent_id` is present, `cy.getElementById(agent_id).flashClass(eventType, 800)` triggers the ping. Datastar handles Mode 1 sidebar content (workspace panels, chat history). The graph canvas is Cytoscape's domain.

### Layout Strategy

- Initial: `cose-bilkent` (fast, organic, good for code graphs)
- Persist node positions to `localStorage` after first stabilization
- Toggle: `dagre` hierarchical layout for file→class→method tree view
- Compound node groups by file path (optional, expand/collapse)

### Suggested Phasing

| Phase | Deliverable | Key Work |
|-------|-------------|----------|
| P0 | Static graph snapshot | `GET /graph/data`, Cytoscape render, no live updates |
| P1 | Live event pings | SSE `/events` → flashClass, live event log sidebar |
| P2 | Mode 1 integration | Sidebar panel, workspace panels, chat (sync) |
| P3 | Chat streaming | SSE token streaming from NodeAgent |
| P4 | Cursor sync | Neovim CursorHold → highlight focused node in graph |
| P5 | Replay scrubber | EventStore replay via timeline slider in Mode 2 |
